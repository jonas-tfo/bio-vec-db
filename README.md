# bio-vec-db

A vector database for amino acid sequences, using embeddings.
Allows nearest-neighbour queries by storing vectors using a HSNW (Hierarchial Navigable Small World)
Storage is backed by [sled](https://github.com/spacejam/sled).
HNSW storage and nearest-neighbour search is backed by [hnsw_rs](https://github.com/jean-pierreBoth/hnswlib-rs).

## Architecture

| Crate | Role |
|---|---|
| `core-db` | HNSW index + sled-backed storage for sequences and embeddings |
| `ml-models` | Calls a Python script to embed sequences via a given transformer |
| `cli` | handles `build` and `query` subcommands |

Embeddings are generated by `scripts/embed_query.py`. The default model is `Rostlab/prot_bert` (1024-dim).

## Requirements
- rust
- uv

For setup, run `uv sync` and make sure **Python 3.13** is downloaded.
This is needed solely for the embedding script.

## Usage

For the concrete flags and descriptions, consult the help page:

**If using cargo:**
```
cargo run -p cli -- --help
cargo run -p cli build --help
cargo run -p cli query --help
```

**If using the binary**
```
cli --help
cli build --help
cli query --help
```

### Examples
#### Build a database from a FASTA file
```sh
cargo run -p cli -- build \
  --fasta sequences.fasta \
  --record-type protein \
  --model Rostlab/prot_bert \
  --dim 1024
```

Use `--fresh` for a rebuild, i.e. to wipe the database before building.

#### Query for nearest neighbours

```sh
cargo run -p cli -- query \
  --query "SAISFHSGYSGLVATVSGSQQTLVVALNSDLGNPGQVASGSFSEAVNASNGQVRVWR" \
  --record-type protein \
  --top-k 5 \
  --output json
```

Output formats: `plain` (default) or `json`.

### Advanced flags: HNSW tuning


| Flag | Default | Effect |
|---|---|---|
| `--ef-construction` | 200 | Index quality (higher = better, slower build) |
| `--ef-search` | 50 | Query accuracy vs speed |


## How does it work
```
data/
  sequence_db/   # sled tree: internal_id → FastaRecord
  vector_db/     # sled tree: internal_id → embedding (f32 vec)
```

### Build
1. The sequences from the fasta are saved in `sequence_db` and individual internal ids are assigned
2. The sequences are embedded and saved in `vector_db` using the same individual internal ids

### Query
1. The HNSW index is rebuilt in-memory from `vector_db` on startup.
2. The k nearest-neighbours are approximated in the HNSW using L2 distance
3. The according internal ids are used to query the according sequences from `sequence_db`
